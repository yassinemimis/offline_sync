Network Layer

The Network Layer is responsible for taking operations out of the Queue Engine and delivering them to the backend. It's pluggable via the SyncTransport contract — Network Adapters like offline_sync_dio are reference implementations, not the only option.

Responsibilities​

  • Translate a queued operation into a concrete network request.
  • Execute the request.
  • Report success, failure, or a conflict back to the sync engine via a SyncTransportResult.

Unlike storage, there's no separate "connectivity gate" that blocks requests while offline — sync() always attempts eligible operations. If the device is genuinely offline, the request simply fails (timeout / no route to host), which the Retry Strategy treats as a normal retriable failure. Connectivity Detection doesn't prevent attempts; it triggers additional sync() calls when the device comes back online, so queued work doesn't sit around waiting for the next manual sync or app restart.

Basic Setup​

core never depends on dio directly — you construct your own Dio instance exactly as you would anywhere else in your app, and hand it to DioSyncTransport:

import 'package:dio/dio.dart';
import 'package:offline_sync_core/offline_sync_core.dart';
import 'package:offline_sync_dio/offline_sync_dio.dart';
import 'package:offline_sync_drift/offline_sync_drift.dart';

final dio = Dio(BaseOptions(
  baseUrl: 'https://api.example.com',
  headers: {'Authorization': 'Bearer <token>'},
));

await OfflineSync.initialize(
  storage: DriftLocalStorage(),
  transport: DioSyncTransport(dio),
);

offline_sync_dio has no opinion on baseUrl, headers, interceptors, or timeouts — all of that is ordinary Dio/BaseOptions configuration, done once, on your own Dio instance:

final dio = Dio(BaseOptions(
  baseUrl: 'https://api.example.com',
  connectTimeout: const Duration(seconds: 10),
  headers: {'Authorization': 'Bearer <token>'},
));

Swapping backends means swapping the transport passed to initialize() — the rest of your app (save, getAll, sync) stays unchanged. A SupabaseSyncTransport / GraphQL equivalent isn't built yet — see Roadmap.

Request Lifecycle​

  1. sync() pulls the next eligible operation off the queue (see Queue Engine).

  2. DioSyncTransport maps SyncOperation.type to a verb against SyncAdapter.endpoint:

    TypeVerbURLBody
    createPOSTendpointoperation payload
    updatePUTendpoint/entityIdpayload + _version (the client's baseline, for optimistic-concurrency checking)
    deleteDELETEendpoint/entityId{ "_version": ... }

  3. The response becomes a SyncTransportResult:

    • Success → operation removed from the queue; the entity is marked synced. See Queue Engine.
    • Retriable failure (timeout, no connection, 5xx) → handed to the Retry Strategy.
    • Non-retriable failure (4xx other than 409) → the operation is marked exhausted once the retry budget — if any — is exhausted. There's currently no callback surfacing this to the app; see Retry Strategy → Giving Up.
    • Conflict (HTTP 409) → the response body is treated as the server's current copy of the entity, handed to Conflict Resolution.

Custom Transports​

Implement SyncTransport directly if your backend isn't REST/Dio-shaped (GraphQL, Firebase, a different verb convention):

class MyTransport implements SyncTransport {
  @override
  Future<SyncTransportResult> send(
    SyncOperation operation,
    SyncAdapter adapter,
  ) async {
    // Inspect operation.type, operation.payload, operation.localVersion,
    // and adapter.endpoint to build your request.
    //
    // Return one of:
    return const SyncTransportResult.success();
    // return SyncTransportResult.failure(retriable: true, message: '...');
    // return SyncTransportResult.conflict(serverData: {...}, serverVersion: 5);
  }
}

offline_sync_dio is the reference implementation, not a required dependency — core only ever depends on the SyncTransport interface.
