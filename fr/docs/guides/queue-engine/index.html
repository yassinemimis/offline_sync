Queue Engine

The Queue Engine is the component responsible for capturing every write operation performed while offline (or online) and persisting it in an ordered, durable queue until it can be safely synchronized with the remote backend.

Why a Queue?​

In an offline-first app, a user action (create, update, delete) cannot always be sent to the server immediately. The Queue Engine decouples when an operation happens from when it is synchronized, guaranteeing that:

  • No operation is lost, even if the app is killed or the device loses power.
  • Operations are replayed in the correct order.
  • Failed operations can be retried without duplicating side effects.

How It Works​

  1. Capture — Every call like OfflineSync.save(user) creates an operation record instead of hitting the network directly.
  2. Persist — The operation is written to local storage (via a Storage Adapter) before returning control to the caller.
  3. Enqueue — The operation is appended to an ordered queue, tagged with metadata (timestamp, entity type, operation type, retry count, and the local version baseline used for conflict detection).
  4. Sync — sync() drains the queue, sending operations through the Network Layer. This runs when you call it manually, opportunistically right after save/delete, or automatically when connectivity comes back (see Connectivity Detection) — there is no persistent background worker yet; syncing while the app is fully closed is planned for a later release (see Roadmap).
  5. Acknowledge — On success, the operation is removed from the queue. On failure, it is handed to the Retry Strategy. On a version conflict, it is handed to Conflict Resolution instead.
// This call is instant and safe, even with no connection.
await OfflineSync.save(user);

Operation Types​

TypeDescription
createA new entity was added locally
updateAn existing entity was modified
deleteAn entity was marked for deletion

Ordering Guarantees​

All queued operations — regardless of which entity they belong to — are processed sequentially, in the order they were created. There is currently no per-entity or configurable concurrency; sync() sends one operation, waits for the result, then moves to the next.

Calling sync() while a sync is already running doesn't start a second, overlapping pass — the second call simply awaits the one already in flight. This means it's safe to call sync() from multiple places (a manual button press, an opportunistic call after save, and an automatic call when connectivity returns) without worrying about them racing each other.

Reading queue status​

Two counters are available, and it's easy to reach for the wrong one:

MethodCountsUse for
OfflineSync.pendingOperationsCount()Operations eligible to be sent right now — excludes failed operations still waiting out a Retry Strategy backoff window.Internal sync logic. Rarely what you want in a UI.
OfflineSync.totalQueuedOperationsCount()Every unsynced operation, including ones currently mid-backoff.UI badges/counters — "3 changes not yet synced". This is almost always the right one for user-facing display.

A device that just failed a send attempt will show pendingOperationsCount() == 0 for the next few seconds (nothing is eligible to retry yet) while totalQueuedOperationsCount() == 1 (the change is still genuinely unsynced). Showing the first number in a UI can misleadingly read as "everything is synced" when it isn't.

final pending = await OfflineSync.totalQueuedOperationsCount();
// e.g. show "3 changes pending" in an AppBar chip

Inspecting the Queue​

A dedicated queue-inspector UI is planned as part of DevTools (not yet available). Until then, use totalQueuedOperationsCount() above for a simple count, or query your configured Storage Adapter directly for full operation details during development.
