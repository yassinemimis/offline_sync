# offline_sync_drift

Drift (SQLite) storage adapter for
[`offline_sync_core`](../offline_sync_core). Implements the `LocalStorage`
contract: the entities table, the sync queue table, and metadata columns
(`isSynced`, `deleted`, `version`).

You don't call anything in this package directly except `DriftLocalStorage()`
— everything else is driven through `OfflineSync` from `offline_sync_core`.

## Install

```yaml
dependencies:
  offline_sync_core: ^0.0.1
  offline_sync_drift: ^0.0.1
```

## Setup

```dart
import 'package:offline_sync_core/offline_sync_core.dart';
import 'package:offline_sync_drift/offline_sync_drift.dart';

await OfflineSync.initialize(storage: DriftLocalStorage());
```

That's it — `DriftLocalStorage()` opens (or creates) a SQLite file named
`offline_sync.sqlite` in the app's documents directory the first time it's
used.

### Generated code

`app_database.g.dart` is generated by `drift_dev` and is not checked into
version control (see the package's `.gitignore`). After cloning or pulling
changes to the table definitions, run this from
`packages/offline_sync_drift/`:

```bash
dart run build_runner build --force-jit
```

> **Why `--force-jit`?** As of this writing, `build_runner`'s default AOT
> compilation step fails on some Dart SDKs with packages that use the
> native-assets/build-hooks feature (`sqlite3_flutter_libs`,
> `path_provider`), with the error `'dart compile' does not support build
> hooks, use 'dart build' instead`. `--force-jit` sidesteps that build step
> entirely. Drop it once upstream fixes this.

## What's implemented (Phase 1)

| `LocalStorage` method | Behavior |
|---|---|
| `saveEntity` | Upsert into `EntitiesTable`, keyed by `(entityType, entityId)`. |
| `softDeleteEntity` | Sets `deleted = true`. Row stays physically present. |
| `hardDeleteEntity` | Physically removes the row. Call once a delete is confirmed synced. |
| `getEntity` / `getAllEntities` | Reads, excluding rows with `deleted = true`. |
| `enqueueOperation` | Inserts a row into `SyncOperationsTable`. |
| `getPendingOperations` | Rows with status `pending` or `failed`, oldest first. |
| `updateOperationStatus` | Used by the (future) network module after a send attempt. |
| `removeOperation` | Called once an operation is confirmed synced. |

`sync()` (in `offline_sync_core`) currently just drains this queue locally
— no HTTP calls yet. See the
[project roadmap](../../README.md#roadmap) for what's next.

## Schema

### `EntitiesTable`

| Column | Type | Notes |
|---|---|---|
| `entityName`\* | `TEXT` | Matches `SyncAdapter.entityName`, e.g. `"User"`. |
| `entityId` | `TEXT` | Matches `SyncAdapter.getId(entity)`. |
| `dataJson` | `TEXT` | JSON-encoded entity (`SyncAdapter.toJson`). |
| `updatedAt` | `DATETIME` | From `SyncAdapter.getUpdatedAt(entity)`. |
| `isSynced` | `BOOLEAN` | True once no pending queue operation references this row. |
| `deleted` | `BOOLEAN` | Soft-delete flag. Excluded from reads when `true`. |
| `version` | `INTEGER` | Bumped on write; will back conflict resolution (Phase 4). |

Primary key: `(entityName, entityId)`.

\* *The SQL column is still named `entityName` — only the Dart-side getter
is called `entityType`, to avoid colliding with a member Drift itself
defines on every generated table class (`TableInfo.entityName`). See
inline comments in `entities_table.dart`.*

### `SyncOperationsTable`

| Column | Type | Notes |
|---|---|---|
| `id` | `TEXT` | UUID, primary key. |
| `entityName`\* | `TEXT` | Which entity type this operation targets. |
| `entityId` | `TEXT` | Which row. |
| `type` | `TEXT` (enum) | `create` \| `update` \| `delete`. |
| `payloadJson` | `TEXT` | JSON snapshot to send. |
| `createdAt` | `DATETIME` | Used to process the queue in order. |
| `status` | `TEXT` (enum) | `pending` \| `failed` \| `synced`. |
| `retryCount` | `INTEGER` | Reserved for Phase 3 (retry/backoff). |

## Testing

```dart
final db = AppDatabase.withExecutor(NativeDatabase.memory());
final storage = DriftLocalStorage(db);
```

`AppDatabase.withExecutor` accepts any Drift `QueryExecutor`, so tests run
against an in-memory database instead of touching disk. See
`test/offline_sync_flow_test.dart` for the full end-to-end example (save →
queue → sync).
