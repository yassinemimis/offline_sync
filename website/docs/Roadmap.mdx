## Detailed Roadmap

### ✅ Phase 0 — Architecture *(Completed)*

- `SyncAdapter<T>`: Contract for connecting user models to the sync engine (`toJson`, `fromJson`, `getId`, `getUpdatedAt`, `endpoint`).
- `SyncOperation` + `SyncOperationType` (`create` / `update` / `delete`) + `SyncOperationStatus`.
- `LocalStorage` (abstract): Contract that every storage implementation must implement.
- `OfflineSync`: Main public API (`initialize`, `register`, `save`, `delete`, `getAll`, `sync`).
- Documentation for five core architectural decisions (`ARCHITECTURE.md`).

### ✅ Phase 1 — Database + Queue *(Completed)*

- `EntitiesTable` + `SyncOperationsTable` (Drift/SQLite).
- `DriftLocalStorage`: Concrete implementation of `LocalStorage`.
- `save()`: Saves data locally immediately and adds an operation to the queue.
- `delete()`: Performs a soft delete and enqueues a `delete` operation.
- `getAll()`: Returns all non-deleted records.
- Successful end-to-end testing.

### ✅ Phase 2 — Network Layer *(Completed)*

> **Goal:** Enable `sync()` to send queued operations over HTTP.

- [`offline_sync_dio`](/docs/adapters/network-adapters/dio) package.
- `SyncTransport` contract (uses `SyncAdapter.endpoint`).
- Automatically selects the appropriate HTTP method based on `SyncOperationType` (`POST` / `PUT` / `DELETE`).
- **On successful request:** `removeOperation`.
- **On failure:** `updateOperationStatus(failed)` instead of removing the operation.

### ✅ Phase 3 — Retry + Backoff *(Completed)*

- Exponential Backoff strategy for failed requests.
- Uses the existing `retryCount` field in `SyncOperation`.
- Configurable maximum retry limit (`maxAttempts`) with timeout and 5xx error handling.
- See the [Retry Strategy](/docs/guides/retry-strategy) guide for complete details.

### ✅ Phase 4 — Conflict Resolution *(Completed)*

- Compare `updatedAt` / `version` between the server and the local device.
- Four conflict resolution strategies built-in:
  - **Server Wins**
  - **Client Wins**
  - **Last Update Wins**
  - **Manual** (Custom developer callback)
- Uses the existing `version` field in `EntitiesTable`.
- See the [Conflict Resolution](/docs/guides/conflict-resolution) guide for complete details.

### ⏳ Phase 5 — Connectivity Detection

- Enable `connectivity_plus` (already included in `pubspec.yaml` but not yet integrated).
- Automatically trigger `sync()` the exact moment network connectivity is restored on the device.

### ⏳ Phase 6 — Background Sync

- Integration with `WorkManager` for background tasks.
- Continue synchronization cycles even when the application is closed or in the background.

### ⏳ Phase 7 — Delta Sync

- Upload and download only changed fields/records (diffs) instead of syncing the entire payload.

### ⏳ Phase 8 — Compression

- GZIP/Brotli compression wrappers for large request payloads to reduce mobile bandwidth usage.

### ⏳ Phase 9 — Encryption

- Encrypt sensitive data stored in SQLite (suitable for healthcare, finance, and other security-sensitive offline apps).

### ⏳ Phase 10 — Logging & Observability

- Structured logs for synchronization events (success, failure, conflict resolutions) with custom logger integrations.

### ⏳ Phase 11 — Alternative Storage Adapters

- `offline_sync_isar`
- `offline_sync_hive`

*Both will implement the same abstract `LocalStorage` contract.*

### ⏳ Phase 12 — Alternative Network Adapters

- `offline_sync_graphql`
- `offline_sync_firebase`
- `offline_sync_supabase`

### ⏳ Phase 13 — Developer Tools

- `offline_sync_cli`: Codegen tool to generate `@SyncModel()` adapters instead of writing `SyncAdapter` implementations manually.
- `offline_sync_devtools`: Flutter DevTools extension tab for monitoring the current synchronization queue and storage live.

### ⏳ Phase 14 — Ready-to-use UI Components

- `offline_sync_ui`: Prebuilt Material and Cupertino widgets such as:
  - Offline status banner/bar.
  - Interactive sync status indicators.

### ⏳ Phase 15 — Example App + pub.dev Release

- Complete `offline_sync_example` demonstrating a real-world multi-entity offline workflow.
- Complete API documentation.
- Publish all monorepo packages to pub.dev.
